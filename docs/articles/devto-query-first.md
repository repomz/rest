> **Write queries. Get an application. Add business logic.**

Starting a new backend usually produces a familiar checklist:

- define models;
- write repositories;
- add services;
- create HTTP handlers;
- introduce request and response DTOs;
- carry database types through every layer;
- describe the API in OpenAPI;
- add tests, configuration, health checks, logging, and Docker.

Then repeat the same process for the next table. And the next one.

At some point, you begin to suspect that a computer could do a meaningful part of this work. The computer, meanwhile, has been using its considerable talents to keep your desk warm.

That observation led to **REST**, an open-source Go CLI that generates a REST application on top of PostgreSQL and SQLC. More important than the tool itself is the idea behind it: **Query-First**.

This article explains the approach, how it differs from spec-first and code-first, where it saves time, and where automation should politely step aside.

## Three possible starting points

### Spec-first

The API contract comes first, commonly as an OpenAPI document:

```text
OpenAPI → server interfaces → implementation → database
```

This works well when the public HTTP contract is the primary artifact shared by multiple teams or external consumers.

However, a specification does not execute a database query. A considerable implementation gap remains between an endpoint description and actual data behavior.

### Code-first

Application code, models, and annotations come first, and the API description is derived from them:

```text
application code → OpenAPI
```

This is convenient when server code is the main source of truth, but persistence models, transport models, and real database operations can gradually drift apart.

### Query-first

REST starts from:

- a PostgreSQL schema;
- named SQL queries;
- types and method signatures generated by SQLC;
- explicit naming conventions.

```text
PostgreSQL schema + named SQL queries
                    ↓
             typed contract
                    ↓
 domain → repository → service → HTTP
                    ↓
       OpenAPI + tests + infrastructure
```

![Query-First flow](assets/query-first-flow.svg)

The schema describes **which data exists**. Named queries describe **which operations the application can actually perform**.

Query-First is not an attempt to make SQL describe every aspect of a production system. It uses an existing executable data contract to generate the large, mechanical part of a backend around it.

## From one query to an HTTP endpoint

Consider a `studies` table:

```sql
CREATE TABLE studies (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    patient         TEXT NOT NULL,
    study_type      TEXT NOT NULL,
    time_beginning  TIMESTAMP,
    surgeon         TEXT NOT NULL,
    dicom_link      TEXT,
    deleted         BOOLEAN NOT NULL DEFAULT FALSE
);
```

Now add a query with optional filters:

```sql
-- name: GetStudies :many
SELECT * FROM studies
WHERE deleted = false
  AND (
    sqlc.narg('date')::timestamp IS NULL
    OR time_beginning::date = sqlc.narg('date')::date
  )
  AND (
    sqlc.narg('type')::text IS NULL
    OR study_type = sqlc.narg('type')
  )
  AND (
    sqlc.narg('surgeon')::text IS NULL
    OR surgeon = sqlc.narg('surgeon')
  )
ORDER BY time_beginning DESC;
```

SQLC turns this into a typed Go method and parameter struct. REST reads the schema, query metadata, and generated signatures, then creates:

```http
GET /studies?date=2026-06-18&type=CT&surgeon=Ivanov
```

Parameters declared with `sqlc.narg(...)` become optional query parameters. Their types flow into Go models and OpenAPI. A `:many` result becomes an array response.

This query:

```sql
-- name: GetStudyByPatient :one
SELECT * FROM studies
WHERE patient = $1 AND deleted = false;
```

becomes:

```http
GET /studies/patient/{patient}
```

And:

```sql
-- name: UpdateStudyDicomLink :one
UPDATE studies
SET dicom_link = $2, updated_at = NOW()
WHERE id = $1 AND deleted = false
RETURNING *;
```

becomes:

```http
PATCH /studies/{id}/dicom-link
Content-Type: application/json

{
  "dicom_link": "https://example.test/studies/123"
}
```

Query names act as a small declarative language:

| Query prefix | HTTP method |
| --- | --- |
| `Get`, `List`, `Find`, `Search` | `GET` |
| `Delete`, `Remove`, `SoftDelete` | `DELETE` |
| `Update`, `Patch` | `PATCH` |
| everything else | `POST` |

Naming conventions are not a complete replacement for an explicit HTTP specification. They are a practical way to derive a predictable contract without describing the same operation in several unrelated files.

## What one command generates

The bundled example can be generated with:

```bash
make build-rest
./bin/rest init --example
./bin/rest gen
go test ./...
```

For an existing SQLC project:

```bash
./bin/rest init
# Configure rest_config/sqlc_rest.yaml
./bin/rest gen
```

The current example produces seven application routes:

```text
POST   /studies
GET    /studies
DELETE /studies
GET    /studies/{id}
DELETE /studies/{id}
GET    /studies/patient/{patient}
PATCH  /studies/{id}/dicom-link
```

It also generates:

- domain models;
- PostgreSQL repositories;
- services;
- HTTP request and response models;
- handlers and handler tests;
- OpenAPI and Swagger UI;
- curl examples;
- a multi-stage Dockerfile;
- optional Zap logging and Prometheus-compatible metrics;
- health checks and graceful shutdown;
- optional Goose initialization migration;
- optional CI/CD workflows.

In a local run, generating the bundled example took roughly **0.6 seconds**, excluding compilation of the CLI itself. This is not a universal benchmark, but it gives the right order of magnitude: less time than it takes to debate whether the next type should be called `StudyResponse` or `StudyDTO`.

## Why the benefit grows with the project

For one table, manually writing the layers may be perfectly reasonable.

The calculation changes with dozens of tables, filters, DTOs, handlers, tests, and OpenAPI operations. Manual effort grows roughly with the number of operations. Generation time barely notices.

Query-First removes several forms of repetition:

1. A PostgreSQL type does not need to be copied independently into repository, domain, transport, and OpenAPI models.
2. Query parameters do not need separate definitions in SQL, handlers, and documentation.
3. Every new named query receives the same architectural structure.
4. Endpoints follow consistent rules instead of the local preferences of whoever wrote a particular file on a particular Tuesday.

The project's main promise is:

> REST turns a PostgreSQL schema and named SQL queries into a REST service ready for further development. In seconds, it creates predictable architecture, type-safe data access, endpoints, OpenAPI, tests, and infrastructure, leaving the developer to implement business-specific logic.

## This is not “business logic generated by magic”

Code generators have a dangerous marketing habit: the demo ends immediately before the first real business requirement appears.

REST does not pretend that a column name is enough to understand a domain. Developers still need to implement:

- complex validation;
- domain rules;
- multi-service workflows;
- unusual response transformations;
- external integrations;
- specialized error behavior.

If a patient must not be scheduled for two procedures at the same time, the PostgreSQL `timestamp` type will not tell us, no matter how thoughtfully we stare at it.

The goal is to remove mechanical work, not engineering judgment.

## Honest limitations

The currently implemented path is **Go + PostgreSQL + SQLC**.

Other current limitations include:

- methods and routes are partly inferred from query names;
- the schema parser does not yet cover every production PostgreSQL construct;
- complex constraints, defaults, and multiple schemas need broader support;
- generated handlers currently use basic status semantics—for example, creation returns `200` rather than a configurable `201`;
- complex validation remains custom code;
- safely evolving an already-generated project remains a hard code-generator problem.

The project includes a `safe_reload` mode. Before regeneration, it compares generated files with the previous snapshot and asks whether user-modified versions should be preserved or overwritten.

That safety net is useful, but a clean boundary between generated and custom code is still more valuable than any recovery mechanism.

### Rate limiting

Rate limiting exists in the planned authentication configuration contract, but it is **not yet implemented** in generated HTTP middleware. It belongs to the roadmap, not the current feature list.

This distinction matters, especially in open source, where every optimistic sentence eventually meets someone with `rg` and a free evening.

## Authentication and authorization: a future chapter

The repository already contains a future auth configuration contract covering identity models, authentication strategies, cookies, refresh tokens, API keys, roles, policies, audit logging, and rate limiting.

The generator itself is not implemented yet.

Once available, this layer should describe:

- whether an endpoint is public or protected;
- required roles and permissions;
- authentication strategies;
- OpenAPI security schemes;
- limits applied to sensitive operations.

This deserves a separate technical article after an end-to-end implementation exists. For now, the honest status is simple: **the contract is designed; implementation is ahead**.

## MongoDB: another future chapter

MongoDB is also represented by a future configuration contract, while generation is not yet implemented.

The interesting design problem is how to preserve the Query-First idea when SQL is replaced by named collection operations, aggregation pipelines, and document validation schemas.

This section should grow into a full chapter once the MongoDB path generates and tests an application end to end.

## Transactions across multiple operations

A single SQLC query maps naturally to one application operation. Real business workflows often need several queries inside one transaction:

```text
create order
→ reserve inventory
→ record payment attempt
→ append audit event
```

A future extension could group named queries into transactional workflows and generate orchestration with `BEGIN`, `COMMIT`, and `ROLLBACK`.

This area requires care because transaction boundaries are part of business semantics, not merely boilerplate. For now, such workflows belong in custom service code. A detailed chapter should follow only after a stable transaction contract exists.

## Query-First does not need to fight Spec-First

Architecture discussions enjoy tournaments. Real projects usually benefit more from cooperation.

A practical workflow could be:

1. Use Query-First to generate a working internal API from real data operations.
2. Refine business semantics and stabilize routes.
3. Share the generated OpenAPI contract with frontend or client teams.
4. Add explicit overrides and compatibility rules before exposing a public API.

Query-First is strongest when data operations are already known and manually building standard backend layers around them adds little business value.

## What I want to learn from the community

The PostgreSQL/SQLC path already works, but the concept is still evolving. I would especially appreciate feedback on:

- Are naming conventions enough, or should method, path, and status use explicit SQL annotations?
- Where should the boundary between generated and custom code live?
- What would a useful transaction workflow declaration look like?
- Which PostgreSQL types and constructs matter most in real projects?
- Should Query-First offer a strict public-contract mode?

Repository: [github.com/repomz/rest](https://github.com/repomz/rest)

If the idea sounds useful, try the bundled example, inspect the generated code, and open an issue with an inconvenient real-world query. Inconvenient queries are remarkably good at teaching generators that life is more complicated than their fixtures.

## Final thought

Query-First does not promise to generate your finished business.

It offers something more practical: take an existing schema and real queries, then build a coherent, type-safe, understandable REST skeleton around them in seconds.

Instead of an empty directory or another abstract tutorial, developers get an application they can run, test, discuss, and gradually turn into a product.

> **Write queries. Get an application. Add business logic.**
